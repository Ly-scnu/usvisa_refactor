# 美签预约系统 usvisa_refactor 项目文件结构说明

> 版本：v0.12  
> 日期：2026-06-17  
> 维护规则：以后新增功能、拆分文件、文件职责变多、模块位置调整，都必须同步更新本文档。  
> 核心目标：保持目录分层清楚，避免二次开发把功能重新堆成一个大文件。

---

## 架构总原则

```text
usvisa_refactor/
|
|-- config/                     # 统一配置文件，前端配置页读写这里
|-- backend/                    # Python 后端，一条龙业务、调度、API、错误恢复
|-- frontend/                   # Vue 前端，总览、诊断、票池分析、配置、清理
|-- storage/                    # 运行时数据、事件、截图、日志、浏览器 profile
|-- docs/                       # 项目说明、交接、开发日志、结构说明
```

分层规则：

```text
00_infrastructure      只能放基础能力，不写抢票业务
01_proxy_management    只处理代理
02_session_context     只处理单轮上下文
03_browser_management  只处理浏览器启动和页面识别
04_snapshot_system     只处理截图和快照稳定性
05_stage_components    只处理一条龙阶段组件
06_business_services   只做业务状态聚合，不直接跑流程
07_scheduler           只做槽位调度和并发控制
08_watchdog            预留守护/卡死检测
09_api_gateway         只提供 API/WebSocket，不写核心流程
99_error_recovery      只放错误识别和恢复组件
frontend/views         只放页面级 UI
```

## 2026-06-14 补充：会话库存水位标准

本项目的智能调度不只看“当前开了几个槽”，还要看槽位是否形成可持续库存。60 秒一次稳定查询的正式标准定义如下：

```text
热查询池 Hot Query Pool       目标 4 个
登录备用池 Login Standby Pool 目标 2 个
候选生产池 Candidate Pool     目标 2 个
普通时段建议活跃槽位           约 8 个，上限仍为 10 个
高峰时段热查询池目标           至少 5 个，候选池至少 3 个
```

### `backend/00_infrastructure/orchestration/sla/session_inventory.py`

会话库存统计模块。只负责分类和统计，不启动槽、不访问官网、不做登录/查询动作。

分类规则：

- `hot_query_sessions`
  - 当前 live 槽位中，已经成功查过 days 的会话。
  - 包括正在冷却的成功会话和已经冷却结束可查询的成功会话。
  - 这是 60 秒轮转的主力池。
- `ready_query_sessions`
  - 热查询池中已经过了单会话冷却、可再次参与查询的会话。
- `cooling_success_sessions`
  - 热查询池中仍在 3-4 分钟冷却的会话。
- `login_standby_sessions`
  - 已到登录/密保边界，但还没有成功查票的备用会话。
  - 用于热查询池失效后快速补位。
- `candidate_sessions`
  - 正在代理、CF、等待室、浏览器启动等前置流程中的候选会话。
  - 用于连续失败或等待室压力升高时替补。
- `terminal_risk_sessions`
  - 命中 1015、access_denied、proxy_banned、browser_crashed 等应回收的风险会话。

输出字段：

```text
inventory.hot_query_sessions
inventory.login_standby_sessions
inventory.candidate_sessions
inventory.hot_deficit
inventory.login_standby_deficit
inventory.candidate_deficit
inventory.health_level
inventory.reason
inventory.slots
```

已接入库存水位：

- 热查询池不足时输出 `inventory_hot_low`。
- 登录备用池不足时输出 `inventory_login_standby_low`。
- 候选池不足时输出 `inventory_candidate_low`。
- 普通库存不足统一输出 `inventory_fill`。
- 决策会把建议槽数抬到库存目标所需的 `desired_inventory_slots`，但仍不超过 `max_slots=10`。

### `backend/07_scheduler/producer_worker.py`

已接入库存水位扩槽：

- `inventory_*` 状态视为库存补位，不再被普通 180 秒扩槽冷却卡死。

### 设计目标

本阶段解决的问题不是“怎么查 days”，而是先保证调度体系不要把明显不能查的会话送进唯一业务 API 闸门，也不要让明显不稳定的代理路线反复浪费槽位。

- `session_health/` 只判断会话当前能不能查；
- `query_gate.py` 只使用健康判断做前置拦截，不写页面/风控分类细节；
- `route_health/` 只维护代理路线健康分数、连续失败和熔断冷却；
